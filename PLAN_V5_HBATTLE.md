# PLAN_V5_HBATTLE.md — 横版战斗施工图（表现层，⏸ 只立项未开工）

> 2026-07-04 用户拍板立项：**战斗加一个横向版本**——现在战场是竖的（我方在下、敌方在上，单位上下走），改成横的（我方在左、敌方在右，单位左右走）。**动机 = 帧动画素材减半**：三国热血画风的角色帧动画按"侧视图"绘制，一套侧视帧 + 水平镜像(flip_h)即可覆盖左右两个朝向；竖版则需要"背面行+正面行"两套行（现 sprite_db 的 row/row_up 结构），美术成本翻倍且侧视最出味。
> **盘面大小、地形、全部战斗逻辑不变——纯表现层改动。** 纵版保留、两版共存可切换（"加一个版本"）。
> **状态：只方案、先记着、未开工**（用户 2026-07-04 指示）。建议开工时机见 §6。

## 0. 铁律（施工时不可违反）

1. **`logic/` 零改动**：10Hz 确定性 tick、arena tile 坐标系（18×32，y 轴=纵深）、lockstep 状态哈希、PvE 重放、AIController 全部不碰。横竖只是"同一逻辑坐标系的两种屏幕投影"。**验收硬标准：同一局重放在纵/横两版下战斗结果与逐 tick hash 完全一致**（反作弊 KAN-78/79 链路天然不受影响，因为 verifier 跑的是 logic 层）。
2. **服务端零改动**、config 数值零改动（可能新增纯表现配置项，如 settings 的版式开关，不入战斗结算）。
3. 两版共存：竖版是已真人验收的基线（KAN-49/S8e 口径都在竖版上），横版没验收通过前不得下掉竖版。

## 1. 现状盘点（写给施工 session，已核实）

- **窗口**：project.godot 720×1280 portrait，stretch=canvas_items，`handheld/orientation="portrait"`。
- **像素账（旋转完美贴合）**：场地 18 宽×32 长（arena.json grid），竖版 720/18=40px/格、1280/32=40px/格 → 横屏 1280×720 下旋转后 32→1280、18→720，**仍是 40px/格，信息密度不变**。河 y∈[15,17]、双桥 x∈[3,5]/[13,15]、部署线 player_y_min=17 / enemy_y_max=15，旋转后河变竖条、桥变上下两座。
- **坐标变换现状**：`view/battle_scene.gd` 用 `_tile_px()` 手算 tile→px 散布 **9 处**（行 264/291/324/358/504/563/605/797/856）；`view/net_battle_scene.gd` 已有**先例**——side2 视角 180° 翻转收敛在 `_t2s()` 一个映射函数 + 少数 flip 感知点（部署 y 界 383 行、精灵朝向 470 行）。➜ 横版照抄这个模式：**把变换收敛成一个函数，再在函数里加旋转分支**。
- **精灵朝向现状**：`view/sprite_db.gd` 每状态 `row`(朝下/正面) + `row_up`(朝上/背面，可选)；battle_scene 按 owner 选行。横版需要"侧视行 + flip_h"新约定；**现有素材没有侧视行**（10 个旧单位是上下行或对称帧）。
- **HUD**：手牌/圣水条/下一张/名片/血条/倒计时按 720 宽竖屏布局，横版需重排到 1280×720。
- **投射物/FX**：涟漪投射物 FX、飘字、DROP_LIFT_TILES（下落起跳偏移，竖向语义）等表现元素存在方向假设，逐个过。

## 2. 横版投影定义（施工基准）

```
竖版（现状）：sx = p.x * tile      sy = p.y * tile        （敌上我下；y 小=敌方）
横版（新）  ：sx = (grid_h - p.y) * tile    sy = p.x * tile  （敌右我左；单位向右=向敌）
net side2   ：先套横版投影，再整场左右镜像（sx' = W - sx），本方永远在左
```
- 我方在左、敌方在右（横版对推惯例，进攻方向=向右与阅读方向一致）。
- 朝向规则：单位移动/攻击目标在其右侧 → 用侧视帧原样；在左侧 → flip_h。Godot 4 用 dst Rect2 负宽或 draw_set_transform 实现镜像。
- 输入逆变换 `_s2t()` 同步旋转（点击/拖拽部署→tile），半场校验用逻辑坐标做（不受投影影响）。

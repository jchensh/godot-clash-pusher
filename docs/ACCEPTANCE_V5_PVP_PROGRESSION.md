# ACCEPTANCE_V5_PVP_PROGRESSION — 真人两机验收用例（KAN-76 养成进 PVP 天梯）

> **验什么**：PVE 练的卡牌养成（等级/阶 → hp/伤害乘区 + 升阶解锁技能）在 PVP 天梯生效，
> 且 **lockstep 不分叉**（全程无 state hash mismatch）。
> 每个用例：**入口 → 操作 → 预期 → 判定**。逐项标 ✅通过 / ❌不通过（附现象）。
>
> 单测已覆盖的部分（不用人验）：乘区曲线数值、服务端下发 level/rank、客户端双方注入、
> 同养成 hash 全等 / 异养成 hash 必分叉。真人验的是**端到端 wire 链路 + 双端视觉一致 + 真实网络下不分叉**。

## 前置（务必照做）

1. **服务器已重建**（本次改了 gateway 的 Go 代码；开发机已 `docker compose up -d --build` 过、healthz 200）。
   若在别的机器部署：`cd server && docker compose up -d --build`。
2. **两台 Windows 各登录不同账号**（下称 **A 机 / B 机**）。两台机 `config/network.json` 指向同一服务端。
3. **拉开养成差距**（A 机操作）：主菜单 → `设置` → GM 工具 → 点 **「全卡满级满阶」**（全卡 10 级 3 阶，乘区 ≈×3.0；
   顺带解锁 golem 且 3 阶 = 亡语召唤已解锁）。**B 机保持白板**（新号或 GM 「重置经济」）。
4. 两机各自在 `卡组` 界面确认天梯卡组（槽 1）；**A 机卡组建议带 golem**（验升阶解锁技能）。
5. 日志观察位：
   - 客户端 Output：`[net][KAN-76] 养成注入: side1[lvlsum=.. ranksum=..] side2[..]` （进房时打）。
   - 服务端：`docker compose logs -f gateway` 看 `mm: room ... prog[lvlsum=.. ranksum=..]`（建房时打）
     和 **不应出现** hash mismatch 相关行。

## 用例 1 — 建房下发：双端日志对上同一份养成

- **入口**：两机主菜单 → `天梯征途` → 各自选卡组槽 1 → 匹配到对方。
- **操作**：进房后看两机 Output 的 `[net][KAN-76] 养成注入` 行 + 服务端 `prog[...]` 行。
- **预期**：① 两机打出的 side1/side2 lvlsum/ranksum **完全一致**（同一份服务端数据）；
  ② 满养成一侧 lvlsum=80 ranksum=24（8 卡 × 10 级 / × 3 阶），白板一侧 lvlsum=8 ranksum=8；
  ③ 与服务端建房日志一致。
- **判定**：✅ 三处日志同一份数据、方向正确（满养成号对应满养成侧）。

## 用例 2（重点）— 数值乘区双端可见：满养成单位明显更肉

- **入口**：接用例 1 的对局。
- **操作**：A（满养成）出一只近战兵（如 knight/giant），让它和 B（白板）的同种兵对撞；两机都观察。
- **预期**：① A 的兵**血条明显更厚、伤害更高**（≈×3.0），1v1 同种兵 A 碾压 B；
  ② **两机看到的战斗过程一致**（谁赢、剩多少血、单位行进节奏——双端渲染的是同一份 sim）；
  ③ 塔血**不**受养成影响（双方塔一样厚，只有出的兵有差距）。
- **判定**：✅ 兵力差距肉眼可见、双端画面一致、塔不缩放。

## 用例 3（重点）— 升阶解锁技能在 PVP 生效：golem 亡语

- **入口**：接同一对局（或再匹配一局）。
- **操作**：A 出 **golem**（3 阶），让它被打死；两机都盯着它死亡的瞬间。
- **预期**：golem 死亡时触发**亡语召唤**（2 阶解锁的 death_spawn，白板 golem 没有）——**两机都看到**召唤物出现、数量一致。
- **判定**：✅ 亡语双端同时触发且表现一致。
- 没带 golem 的替代验法：archers 3 阶 = **3 个弓手**（白板 2 个）——A 出 archers 数一下人头。

## 用例 4（命门）— 全程 lockstep 不分叉 + 正常结算

- **入口**：把用例 1~3 的对局**完整打完**（正常打到王塔破/超时，别中途退）。
- **操作**：打完后查服务端日志：`docker compose logs gateway | grep -i "mismatch\|diverg"`。
- **预期**：① **零输出**（全程两端 hash 一致——养成注入没有破坏确定性）；
  ② 对局正常结算：胜负弹窗、杯数/MMR 变动入库（可再匹配一局看杯数变过）；
  ③ 对局中双端无卡顿/单位瞬移/回滚等 desync 视觉症状。
- **判定**：✅ 无 mismatch 日志 + 正常结算。**这条不过 = 整个功能不过**，回报我修。

## 用例 5 — 断线重连不丢养成

- **入口**：再开一局（A 满养成 vs B 白板）。
- **操作**：开打约 30 秒后，**A 机断网 5~10 秒再恢复**（拔网线/关 WiFi），等自动重连回局。
- **预期**：① A 重连回到当前进度（重放快进）；② 重连后 A 再出的兵**仍是满养成数值**（JoinRoomResp 重放带 progress）；③ 继续打完无 mismatch。
- **判定**：✅ 重连后养成仍在、结算正常。

## 用例 6 — 白板对白板：零回归兜底

- **入口**：B 机再找一个白板号（或 A 机 GM 重置经济后），两个白板号打一局天梯。
- **操作**：正常打一局。
- **预期**：与改动前的天梯完全一致（双方 lvlsum=8 ranksum=8，数值全 baseline），无任何异常。
- **判定**：✅ 白板局行为不变。

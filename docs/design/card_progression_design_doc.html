卡牌升级 / 升阶系统

为「对推小游戏」提供贯穿全生命周期的单卡成长深度：以金币驱动高频升级、以碎片门控深度升阶，把关卡产出回收成战力增长，形成「打关 → 攒资源 → 变强 → 打更深的关」的核心养成闭环。

1设计目标与定位

先回答"为什么要有这个系统"——它服务的玩家体验与商业目标，是后面所有数值与交互取舍的裁判。

1.1 为什么存在

  • 长期成长目标：给玩家一条清晰、可预期、永远差一点点就能到手的变强路径，是 PvE 闯关与留存的主发动机。
  • 货币回收池：承接关卡/挂机产出的金币与定向掉落的碎片，让产出有去处，维持经济健康。
  • 付费锚点：养成深度=时间成本，为后续 IAP（金币礼包/碎片直购/加速）预留天然付费点（本案只做养成框架，具体商业化按上线节奏推后）。
  • Build 表达：升阶解锁的技能积木（+数量 / 减速 / 亡语裂变…）让"练哪张卡"成为策略选择，而非纯数值堆叠。

1.2 三条养成线一览

本系统由三条相互咬合的线组成，共享稀有度定价，但资源与手感各不相同：

线做什么消耗收益节奏
解锁攒够该卡碎片，把卡从"未拥有"变为"可用"碎片（定向）获得新卡 / 新 Build里程碑
升级在当前阶的上限内逐级提升等级金币（通用）全属性 +10%/级（线性）高频、小额
升阶突破，抬高等级上限 + 属性乘区 + 解锁技能碎片 + 金币（大额）×1.25 乘区 + 上限+3 + 技能积木低频、里程碑
设计意图 —— 双货币分工。金币是流量货币（高频、小额、通用），负责让玩家几乎每天都能点到"变强"；碎片是稀缺定向货币（每卡独立），门控深度养成与解锁，制造"想练这张就得针对性攒"的取舍。二者叠加，既有即时反馈又有长期目标。

1.3 设计原则（铁律）

服务器唯一权威（决策 48）。等级/阶/碎片/金币/解锁状态全部由服务器计算并落 PG 库，客户端只是瘦表现层 + 只读缓存。所有升级/升阶/解锁/领取动作走服务器 API 校验，改本地存档或改客户端时钟一律无效；断线即不可养成（fail-closed）。
配置化，不硬编码。成长曲线、成本、上限、稀有度定价全部读 config/economy.json 与 config/card_progression.json，数值可由策划改表即调，程序不写死任何数字。
稀有度 ≠ 强度。稀有度只决定养成成本与获取难度（碎片/金币定价、掉率），不等于满养成后更强——低稀有度卡满养成同样应有一线之地，避免"高稀有度即答案"。

3系统总览与经济闭环

一张图看清系统在整个游戏经济里的位置：它既是产出的回收口，又是战力的生产口。

闭环健康度是本系统的第一 KPI。产出（关卡+挂机）必须与消耗（升级+升阶+解锁）在目标付费/时长曲线上大致配平：产出过剩 → 玩家秒满、失去目标；产出不足 → 卡点劝退。配平参数与曲线在 §5，最终值由 V5-S8 balance probe 标定。

4核心规则

系统主体。每条规则都写成"可翻译成 if…then… 和一条测试用例"的粒度。

4.1 解锁（Unlock）

  • 每张未拥有的卡有一个解锁碎片阈值 unlock_shards[rarity]：普通30 / 稀有50 / 史诗80 / 传说120。
  • 玩家通过关卡定向掉落/商店/活动累积该卡碎片；当 shards[card] ≥ 阈值 时，"解锁"按钮可点。
  • 点击解锁 → 服务器扣除该阈值数量碎片，置 unlocked=true，卡初始化为 阶1 / 等级1。
  • 解锁后剩余碎片保留，可继续用于后续升阶。
  • starter 卡（starter=true，如 knight/archers/giant…）默认已解锁，跳过本步。
规则示例（可测）：if 未解锁 and shards[card] ≥ unlock_shards[rarity] → 显示"解锁"按钮；点击后 shards -= 阈值，unlocked=true，level=1,rank=1。测试用例见 §11。

4.2 升级（等级线）

  • 升级只花金币，一次 +1 级，无失败、无随机。
  • 可升级的前提：当前等级 < 当前阶的等级上限（见 level_cap_per_rank）。达到上限时升级按钮转为"需升阶"引导。
  • 每级带来线性属性增幅：全属性（攻/血/等）× (1 + 0.10 × (等级-1))。第10级即基础的 1.9 倍。
  • 升级金币成本随等级线性递增（非指数，刻意压平曲线，对齐皇室战争 2025 改版方向），公式与全表见 §5.2。
为什么升级只花金币、且成本线性。升级是玩家的"日常爽点"，要高频、可预期、无挫败。金币来源稳定（关卡+挂机），线性成本让玩家清楚"再打几关就能升"，避免指数墙带来的劝退（这正是皇室战争把成本从指数改线性的动因）。

4.3 升阶（突破线）

  • 升阶是里程碑动作，共 3 阶（初始阶1，可升到阶3，即两次升阶）。
  • 升阶前提：当前等级已达当前阶的上限（阶1须到 Lv4、阶2须到 Lv7 才能升阶）——先满级、再突破，节奏张弛有度。
  • 升阶消耗 碎片 + 金币（大额，见 rank_up 表）。
  • 升阶带来三重收益：
    1. 抬高等级上限：阶1→2 上限 4→7；阶2→3 上限 7→10（突破式解锁，对齐原神"突破抬上限"）。
    2. 属性乘区：× 1.25 的整卡乘区，按 1.25^(阶-1) 叠乘（阶3 即 ×1.5625）。
    3. 解锁一个节点 rank_unlocks[rank]，二选一类型：
类型含义示例（真实配置）
stat 数值跳纯数值台阶，由属性乘区吸收，无机制变化（无 ops）骑士 knight 阶2「数值跳」；步兵/枪兵多为此型
skill 技能积木改/加战斗机制，带 ops，由 effective_skills 应用弓手 archers 阶3「+1 弩手 (2→3)」count_add:1；火球阶3「伤害+60」；金身罗汉 golem 阶3「裂物变石心攻城兽」unit_field
已知实现边界（须在文案/UI 标注）。部分"新机制"技能积木（on-hit 溅射/连锁/穿透/灼烧/护盾/DPS 递增等）当前战斗引擎尚未支持，配置里仅以 note 占位、留 V5-S8 / KAN-88 实装。这些卡的高阶节点在实装前应降级为数值跳或标注"敬请期待"，不可让玩家花了资源却无实际效果。逐条见 card_progression.json 的 note。

4.4 属性最终计算（三线合成）

# 单卡某属性的最终值（战斗读取值） 最终属性 = 基础属性 × (1 + level_stat_per_level × (level - 1)) # 等级线：线性，0.10/级 × rank_stat_mult ^ (rank - 1) # 升阶线：乘区，1.25^(阶-1) × Π(技能积木 num_mult / +count / …) # 技能线：由 effective_skills 施加

满养成（阶3·Lv10）纯数值倍率 = 1.90 × 1.5625 ≈ 2.97× 基础（未计技能积木额外收益）。

6.1 服务器持久化（权威）

表/字段类型说明

6.2 配置源（服务器权威源，登录下发带版本配置包）

config/economy.json 成长曲线/成本/上限/挂机/奖励 config/card_progression.json 每卡稀有度/base_power/rank_unlocks(ops) config/cards.json 卡基础定义 config/units.json 单位基础属性（技能积木作用对象）

6.3 API 契约（示意）

接口请求服务器校验 & 结果
POST /card/unlock{card_id}校验 shards≥阈值 → 扣碎片、置 unlocked、初始化 lv1/r1 → 返回新态 + 钱包
POST /card/upgrade{card_id}校验 已解锁 & lv<当前阶上限 & gold≥cost(lv) → 扣金币、lv+1 → 返回新态
POST /card/rankup{card_id}校验 lv=当前阶上限 & rank<3 & shards≥ & gold≥ → 扣碎片+金币、rank+1 → 返回新态 + 解锁节点
 所有写接口须幂等（携带 client_op_id 去重）、fail-closed（校验不过原样驳回、不改态），响应统一带最新钱包与卡态，客户端据此刷新缓存。

7.3 反馈与红点

  • 正反馈：升级/升阶成功 → 数值绿色跳字 + 战力+X 飘字 + 音效；升阶额外播突破特效 + 阶星点亮。
  • 红点规则（红点是负反馈，只挂核心收益点）：某卡金币&碎片均足够可升级/升阶/解锁时，在卡入口挂红点；不满足则不挂。避免"永远有红点"的焦虑。
  • 预览：hover/选中升级按钮时，属性行实时预览升级后数值（▲差值），点前可见收益。
交互对齐项目框架：升阶确认、资源不足提示等覆盖层必须经 UI.modal() 入弹窗层（CanvasLayer 50），跳字/toast 走 UI.toast()（90层不挡手）；切页走 Router.goto()；刷新订阅 Events.economy_changed，动作 handler 不手动重刷。

8边界与异常

系统的专业度在边界。以下每条都须有明确处理，不能"看情况"。

情形处理
金币 / 碎片不足按钮置灰 + 缺口数值提示 + 一键跳转获取途径；服务器兜底二次校验驳回
已达当前阶等级上限升级按钮转"去升阶"引导；直接调 upgrade 接口须被服务器驳回
阶3 满级 / 阶3 再升阶升级"已满级"置灰；升阶按钮隐藏或置灰，rankup 接口驳回
未解锁却尝试升级/升阶接口驳回；UI 不应暴露入口
断线 / 请求超时fail-closed：不本地假装成功，按钮回滚可点态，toast"网络异常,请重试"；重连后以服务器态为准覆盖
并发重复点击（连点两次升级）接口幂等（client_op_id 去重）；客户端点击后锁按钮至响应返回
配置版本不匹配登录下发配置版本；客户端缓存版本 ≠ 服务器 → 强制拉取新配置，禁止用旧曲线计算
技能节点引擎未支持该阶节点在实装前降级为数值跳 / 标注"敬请期待"，不得扣资源却无效果（见 §4.3 风险）
金币溢出上限钱包设上限（如 int64 内的软上限）；达上限时挂机停止累积并提示"仓满"

9跨系统接口

系统关系
关卡 / 挂机上游产出：向钱包发金币、向 shards 发定向碎片（首通/重复/离线结算，服务器算）
战力系统下游消费：队伍战力 = Σ(base_power × 养成系数)，养成系数由 level/rank/技能折算；升级升阶后触发战力重算
战斗 / 卡组下游消费：出战时 CardProgression.effective_skills 把 rank_unlocks.ops 施加到 units.json 基础属性，得到本局单位实际数值/机制
事件总线任何养成态变更 → 服务器返回新态 → EconomyStateCache 收口发 Events.economy_changed → 卡列表/详情/战力页统一刷新
商店 / 活动 / IAP上游产出：直购金币/碎片、碎片礼包（商业化按上线节奏，本案预留接口不实做）
反作弊所有产出/扣费经服务器；PVE 结算受 min_stage_duration_s / max_cmds_per_battle / 重放抽验 约束（KAN-78/79）

10埋点与数据分析

让系统可迭代——上线后靠这些数据判断配平健康、定位卡点。

事件埋点

  • card_unlock：card_id, rarity, 累计碎片耗时
  • card_upgrade：card_id, from_lv→to_lv, gold_spent, 钱包余额
  • card_rankup：card_id, from_rank→to_rank, shards_spent, gold_spent, 解锁节点
  • upgrade_blocked：因金币/碎片/上限被挡的次数与情形（劝退信号）

健康度看板指标

  • 卡点分布：玩家最多停留在哪个稀有度/等级/阶（暴露成本墙）
  • 金币/碎片盈余分布：产出是否过剩或枯竭（配平第一信号）
  • 各卡养成渗透率：哪些卡被普遍练、哪些无人问津（掉落/强度问题）
  • 首次升阶转化率：新手是否顺利跨过第一次突破（教学/成本问题）

11验收标准

一条条可勾选，QA 与策划共同据此判定"算不算做完"。

  • 解锁：碎片达阈值前按钮不可点；达阈值点击后碎片正确扣减、卡初始化为阶1/Lv1、剩余碎片保留。
  • 升级：金币足且未达上限时 +1 级；属性按 (1+0.10×(lv-1)) 正确变化；金币按 §5.2 表精确扣减。
  • 上限门控：阶1 到 Lv4 后升级按钮转"去升阶"；直接请求 upgrade 被服务器驳回。
  • 升阶：满足"满级+资源足"才可升；升阶后上限 4→7/7→10、乘区 ×1.25 生效、对应 rank_unlock 节点解锁。
  • 技能积木：archers 阶3 战斗中弩手 2→3；火球阶3 伤害+60；引擎未支持节点按降级策略处理、不空扣资源。
  • 权威性：改本地缓存/改客户端时钟均无法越权升级；断线时不产生本地假成功，重连以服务器态覆盖。
  • 幂等：连点两次升级只生效一次，金币只扣一次。
  • 反馈：升级/升阶成功有数值跳字+战力更新+音效；升阶播突破特效+阶星点亮。
  • 红点：仅在"可升级/升阶/解锁且资源足"时出现，条件消失即消。
  • 配置化：改 economy.json 任一数值后，无需改代码即在下次登录生效。
  • 刷新：任一养成动作后，卡列表/详情/战力页经 economy_changed 自动一致刷新，无需手动重进。

## 0. 总原则（决策 48，永久）

**服务器唯一权威**：账号/钱包/养成/王国/进度/配置全部服务器算 + PG 落库、服务器时钟结算；
客户端是瘦表现层，改本地存档/缓存/时钟一律无效。客户端能骗的只有它自己的屏幕。

## 1. 三套子系统，三种防法

| 板块 | 战斗在哪算 | 防作弊思路 |
|---|---|---|
| 资源/养成/王国 | 不涉战斗，纯服务器结算 | 服务器时钟 + 全量校验（成本/上限/门槛），客户端只发意图 |
| PVP（lockstep） | 两台客户端各算一遍 | 双端 + 指纹对帐：**两个考生互相对答案** |
| PVE（闯关） | 单台客户端算 | 证据链 + 事后重放：**单人考试全程录像，考官重演验卷** |

### 1.1 资源 / 挂机 / 王国计时

- 挂机金币（K3 起=铸币坊）、建筑施工、加速定价：全部以 DB 里的**服务器时间戳**为准，
  领取/完工时服务器用自己的 `now` 算差值。客户端倒计时纯显示。
- 所有产出/扣费/升级走 `/v5/*` API 服务器校验（资源够不够/王城门/章节门/工匠队占用）。
- 商业化铁门：金币买不到城建资源（配置校验双端锁死，`kingdom.ParseConfig` + ConfigLoader）。

### 1.2 PVP（V4 lockstep）

- 服务器**不跑战斗模拟**（省算力的架构选择），两端各自跑同一套确定性 logic/，
  每隔 HASH_EVERY tick 上报状态指纹（state hash）由服务器对帐；分叉=判 desync/作弊。
- 一切进战斗的数值由服务器权威下发、两端对称注入：卡牌 level/rank（KAN-76 `CardProgress`）、
  王国城防（K5 `TowerBonus`）。两端不各自推导，杜绝"我端悄悄改数值"。
- **每新增一个进战斗的数值维度，必须交两条 lockstep 测试**（`test_lockstep_determinism`）：
  ①同值同输入逐 tick 哈希全等（管线确定性）②异值同指令哈希必分叉（对帐网罩得住新维度）。
  KAN-76 与 K5 均已交付同款；未来宫廷武将等照办。
- 确定性红线：10Hz 固定 tick、无随机、禁 pow（KAN-77 改循环乘法）、坐标毫 tile 整数编码。

### 1.3 PVE（KAN-78/79 证据链 + verifier 重放）

四道锁：
1. **开战报到**（`pve_start`）：服务器时钟记 `started_at`，从 DB 读权威养成快照
   （level/rank + K4 城防 `_towers` 保留键）发 `battle_id`。没报到没法领奖。
2. **过程直播上传**（`pve_report`）：指令流 + 周期 hash 分批上传，服务器按**到达墙钟**记批次
   ——想交出一局 3 分钟的证据必须让 3 分钟真实流逝。
3. **结算 sanity**（`stage-clear`）：battle 属己/关匹配/未消费（防重放领两次）/
   墙钟 ≥ 最短时长/**声称 ticks÷10 ≤ 墙钟×1.15+5**（时间压缩必拒）/星数与摘要自洽。
4. **事后重放**（verifier 容器，乐观发奖后验）：用你上传的指令流 + **服务器记录的**养成/城防
   快照，同一套 logic/ 重演整局、逐段核对 hash。对不上 = 战果无法从自己证据复现 →
   mismatch → 账号影子标记（验证期只标记不回滚）。改本地缓存刷满级打的局在这里现形
   （`test_tampered_progress_forks` / `_towers` 缺键分叉均有测试实证）。
- ⚠️ 运维铁律：verifier 挂载工程 `logic/`+`config/`，改这两处必须 `docker restart server-verifier-1`。

## 2. 加速器（变速齿轮类外挂）风险盘点（2026-07-19）

| 攻击面 | 结论 | 依据 |
|---|---|---|
| 刷资源/跳施工计时 | **免疫** | 全部服务器时钟结算，客户端倒计时纯显示 |
| PVP 变速 | **免疫** | 节拍=服务器广播 TickBundle 驱动，客户端只能等包；改出的数值差被 hash 对帐抓 |
| PVE 加速（压缩时长） | **当场拦截** | `ticks÷10 ≤ 墙钟×1.15+5` + 墙钟下限 + 批次到达时间记账 |
| **PVE 减速/暂停** | ⚠️ **残余风险（低危）** | 见下 |

### 2.1 残余风险：PVE 减速

墙钟校验刻意单向（≤）——切后台/掉线/卡顿会让墙钟远大于战斗时长，反向硬拒会误杀正常玩家。
因此开减速器（如 0.5x）打 PVE：每战斗秒有更多真实时间做微操 → 打星变容易；
重放验证器抓不到（证据自洽，只是"打得慢"）。

**危害评估：低**——只影响 PVE 打星难度；无资源无限刷口子；不影响经济安全与 PVP 公平。

**处置：挂账 [KAN-113]（待办，以后优化）**。对策选项（做时选型，建议先 ② 纯日志观察零误杀）：
① 墙钟/tick 宽松上界比率（K 需真实数据标定，容忍断线豁免窗）；
② `pve_report` 批次节奏分析（到达间隔 vs 批内 tick 跨度均匀拉长 = 减速特征）；
③ 与 verifier 影子标记联动，观察口径跑一段再定判罚线。

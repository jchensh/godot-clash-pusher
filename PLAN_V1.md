## 0. 给 Agent 的开发纪律（最高优先级，先读）

**必须遵守：**
- **一步一确认**：严格按「第六节 开发步骤」顺序执行。**每完成一步，先停下等用户确认，再进入下一步。** 不要一次性把多步做完。
- **每步一次 git commit**：每步完成且通过验收后提交，commit message 描述本步内容（例：`step2: elixir system + unit tests`）。
- **逻辑先行、测试覆盖**：第 0~6 步是纯逻辑层，**必须配单元测试**，测试通过才算完成。
- **给成功标准而非实现细节**：每步有明确「验收标准」，以通过验收为目标，不要自行扩大范围。

**禁止事项（DO NOT）：**
- ❌ **不要用物理引擎做 lane 碰撞**。不要用 `RigidBody2D` / `Area2D` 物理碰撞处理单位互撞。lane 推进碰撞用「队列前后关系」的纯逻辑判断即可。
- ❌ **逻辑层不要依赖屏幕像素坐标**。单位位置一律用 `0.0~1.0` 的 lane 进度表示（0=己方塔，1=敌方塔）。
- ❌ **不要把游戏速度绑定渲染帧率**。圣水/时间/推进用固定逻辑 tick 结算，显示层做插值。
- ❌ **不要过度设计**。发现自己在写超长、超复杂的抽象时，停下来反问「能不能更简单」。不写用不到的「未来扩展」代码。
- ❌ **不要擅自删改与当前任务无关的代码或注释。**
- ❌ **不要替用户做隐含假设就往下跑**。遇到配置字段含义不清、接口不明确时，**先提问**，不要猜。

## 1. 项目一句话

竖屏单机游戏：玩家 vs 规则 AI，双方用**圣水 + 循环卡组**部署单位，单位沿固定 lane 互相推进，目标是推掉对方的塔。核心是**皇室战争的对推/出牌博弈机制**（塔是胜负目标，不是塔防玩法主体）。先在 Windows 跑通，再导出安卓。V1 纯 2D、纯几何白膜、无服务器、无防作弊。

## 2. 设计基线（已锁定，不得擅自更改）

| 维度 | 决定 |
|---|---|
| 核心玩法 | 皇室战争对推机制（圣水 + 循环卡组 + 单位互推 + 三塔制） |

| 美术 | V1 纯几何白膜（ColorRect / Polygon2D），后期换皮 |
| 平台 | 先 Windows 跑通，再导出安卓 APK |
| 引擎 | Godot 4.x |

## 3. 核心架构原则：逻辑层 / 显示层彻底分离

**逻辑层**持有游戏的真实状态（谁在哪、多少血、多少圣水），完全不关心画面长什么样。
**显示层**只负责把逻辑状态画出来，每帧读取逻辑状态并更新视图。

带来三个好处：
1. **白膜换皮零成本**：换美术只改显示层，逻辑层不动。
2. **玩家与 AI 天然对称**：两者都只是「向逻辑层发指令」，逻辑层不在乎指令来自触摸还是 AI。
3. **为未来联机/防作弊留门**：逻辑层独立，未来可搬到服务器。

**数据流：**
```
玩家点击 → InputHandler → Player.出牌() → SkillSystem 生成单位
                                              ↓
                          逻辑层每个逻辑 tick update（推进、碰撞、扣血）
                                              ↓
                          BattleScene 读逻辑状态 → 更新所有 View（含插值）
```
AI 线相同，仅把「玩家点击 → InputHandler」替换为「AIController 决策」，其余复用。**这就是对称性在代码中的落地。**

## 4. 模块拆分

### 逻辑层（不依赖 Godot 渲染）
| 模块 | 职责 |
|---|---|
| `Battle` | 战斗总控：对局计时、双方状态、胜负判定 |
| `Player` | 一方状态：圣水、手牌、牌库循环。**玩家与 AI 共用** |
| `Elixir` | 圣水随时间回涨、出牌扣除、封顶 |
| `Deck` | 8 张牌库 + 4 张手牌的**循环队列**（出一张从库尾补一张） |
| `Unit` | 场上单位运行时状态：血、lane 进度、所在 lane、攻击目标 |
| `Lane` | 一条 lane 的单位列表、推进与碰撞结算 |
| `Tower` | 三塔血量与摧毁判定 |
| `SkillSystem` | 解析配置中的技能积木并执行（生成/直伤/AOE） |
| `ConfigLoader` | 读取 JSON 配置并转为游戏内数据 |

### AI 层
| 模块 | 职责 |
|---|---|
| `AIController` | 观察战局 + 圣水，按规则决定何时出哪张牌，向 `Player` 发指令 |

### 显示层（Godot 节点）
| 模块 | 职责 |
|---|---|
| `BattleScene` | 主场景，每帧把逻辑状态画出来 |
| `UnitView` | 单个单位的可视化（V1 = 带颜色的方块/圆/三角） |
| `CardUI` | 底部 4 张手牌按钮 |
| `ElixirBarUI` | 圣水条 |
| `TowerView` | 塔显示 + 血条 |
| `InputHandler` | 接收触摸/点击，转成「在 lane X 出第 N 张牌」指令 |

**建议目录结构：**
```
/logic    逻辑层脚本
/view     显示层脚本与场景
/ai       AIController
/config   cards.json / units.json / levels.json
/tests    单元测试
```

## 5. 配置表设计（全 JSON）

### 5.1 卡牌 `cards.json`
一张卡 = 圣水消耗 + 一组技能积木数组（`skills` 是数组，可叠多个积木）。
```json
{
  "knight": {
    "name": "骑士",
    "elixir_cost": 3,
    "skills": [
      { "type": "spawn_unit", "unit_id": "knight_body", "count": 1 }
    ]
  },
  "fireball": {
    "name": "火球",
    "elixir_cost": 4,
    "skills": [
      { "type": "aoe_damage", "radius": 1.5, "damage": 300 }
    ]
  },
  "arrows": {
    "name": "射箭",
    "elixir_cost": 3,
    "skills": [
      { "type": "direct_damage", "damage": 150, "target": "first_enemy_in_lane" }
    ]
  }
}
```

### 5.2 单位 `units.json`
`spawn_unit` 生成的实体定义。卡与单位拆开，因为一张卡可能生成多个单位。
```json
{
  "knight_body": {
    "name": "骑士",
    "hp": 600,
    "damage": 75,
    "attack_speed": 1.2,
    "move_speed": 1.0,
    "attack_range": 0.5,
    "target_type": "ground"
  }
}
```

### 5.3 关卡/对局 `levels.json`
```json
{
  "level_01": {
    "name": "新手关",
    "elixir_regen_rate": 1.0,
    "elixir_max": 10,
    "match_duration": 180,
    "player_deck": ["knight", "fireball", "arrows", "..."],
    "ai_deck": ["knight", "arrows", "..."],
    "ai_difficulty": "normal",
    "tower_hp": { "king": 2400, "princess": 1400 }
  }
}
```
**要点**：圣水回速、对局时长、塔血量全走配置，不改代码即可调出不同节奏的模式。

## 6. 开发步骤（施工图，按序执行，每步停下确认 + commit）

| 步骤 | 内容 | 验收标准（成功条件） |
|---|---|---|
| **0** | 项目脚手架 + Godot 工程结构 + git 初始化 | Godot 能打开空工程，目录分好 logic/view/config/ai/tests |
| **1** | `ConfigLoader` + 三张 JSON 配置 | 测试能把 cards/units/levels 读入内存并打印验证 |
| **2** | `Elixir` 圣水系统 + 单元测试 | 圣水按配置速率回涨、出牌扣除、封顶正确；时间走逻辑 tick |
| **3** | `Deck` 循环抽牌 + 单元测试 | 8 库 + 4 手、出一张补一张的循环正确 |
| **4** | `Unit` + `Lane` 推进与碰撞（纯逻辑，无画面） | 测试：两单位在 lane 相遇互相掉血、死亡移除；位置用 0~1 进度 |
| **5** | `Tower` + `Battle` 胜负判定 | 单位推到底扣塔血，塔血归零判胜负 |
| **6** | `SkillSystem` 三积木实现 | 出一张卡能正确触发 生成/直伤/AOE |
| **7** | 显示层：白膜方块 + 手牌 UI + 圣水条 + 血条 | **MVP！** 玩家能用手点着出牌、看单位推进、推塔、出胜负 |
| **8** | `AIController` 规则 AI 接入 | AI 自己出牌对抗，一局正常分胜负 |
| **9** | 安卓导出 + 触摸适配 + 竖屏配置 | Windows 上导出 APK，手机能跑能玩 |

**顺序理由**：第 0~6 步全是无画面的纯逻辑，可用单元测试保过；第 7 步接显示层时逻辑已稳，出问题能立刻分辨「逻辑错还是显示错」。
**MVP 节点 = 第 7 步**（先用 1 条 lane 跑通）。lane 从 1 条扩到 3 条，在第 7、8 步之间灵活插入。

## 7. 资源清单（白膜阶段不做，先列出备查）

白膜阶段全部用 Godot 内置 `ColorRect` / `Polygon2D` 画几何图形（近战=方块、远程=圆、建筑=三角），零美术成本。玩法验证后再按下表找素材（推荐 Kenney.nl，免费可商用）或 AI 生成。

**视觉**
- 单位：每种 1 待机 + 1 攻击表现
- 塔：国王塔 + 公主塔（含血条）
- 子弹/投射物：远程与 AOE 飞行物
- 卡牌图标：每张卡一个
- 战场背景：竖屏战场 + 3 lane 分隔 + 中间分界线
- UI：圣水条、血条、手牌框、胜负结算界面、主菜单

**音频**（V1 可省略）
- 出牌、单位攻击、塔被击中、胜利/失败音乐

**字体（重要）**

## 8. 已预判的坑（开发时盯紧）

1. **lane 位置用 0.0~1.0 进度**，不用像素坐标（0=己方塔，1=敌方塔）。显示层再翻译成像素。否则换分辨率/上手机就乱。
2. **碰撞用 lane 队列逻辑判断**，不用物理引擎。模型常「热心」上 `RigidBody2D`，是过度设计，禁止。
3. **圣水与时间用固定逻辑 tick**（如每 0.1 秒结算一次），不绑渲染帧；显示层做插值。第 2 步就要定好。
